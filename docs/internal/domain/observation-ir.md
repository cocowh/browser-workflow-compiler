# Observation IR

## Purpose

Observation IR records what happened during a browser recording session.

It is the factual source for later analysis. The system should preserve observations before trying to explain or simplify them.

## Design Principles

### Facts before interpretation

Observation IR should record raw events and evidence. Analysis such as action-request linking or field dependency detection should be stored separately.

### Stable event envelope

All event types should share a common envelope so they can be stored, queried, sorted, and replayed through the same pipeline.

### Artifact references instead of large inline payloads

Large bodies, screenshots, DOM snapshots, and traces should be stored as artifacts and referenced by ID or URI.

### Privacy aware

Observation IR may contain sensitive headers, cookies, tokens, or request bodies. P0 should at least make this visible in the model so redaction can be added later.

## Common Event Envelope

```yaml
id: evt_001
sessionId: sess_001
type: browser.click
timestamp: 1781190000000
sequence: 1
pageUrl: https://example.com/cart
frameId: main
actor: user
payload: {}
artifactRefs: []
tags: []
```

## Common Fields

- `id`: stable event ID.
- `sessionId`: recording session ID.
- `type`: event type.
- `timestamp`: event timestamp.
- `sequence`: monotonic ordering inside the session.
- `pageUrl`: page URL when the event occurred.
- `frameId`: browser frame ID when available.
- `actor`: user, browser, network, worker, analyzer, or replay.
- `payload`: event-specific data.
- `artifactRefs`: references to bodies, screenshots, DOM snippets, or trace files.
- `tags`: optional labels such as sensitive, noise, graphql, auth, static-asset.

## P0 Event Types

### Browser events

- `browser.session_started`
- `browser.session_stopped`
- `browser.navigate`
- `browser.click`
- `browser.input`
- `browser.wait`
- `browser.screenshot`
- `browser.dom_snapshot`

### Network events

- `network.request`
- `network.response`
- `network.request_failed`
- `network.graphql_operation`

### Storage events

- `storage.cookie_snapshot`
- `storage.local_storage_snapshot`

### Replay events

- `replay.started`
- `replay.step_result`
- `replay.finished`

## Example: Click Event

```yaml
id: evt_010
sessionId: sess_checkout
type: browser.click
timestamp: 1781190001000
sequence: 10
pageUrl: https://shop.example.com/cart
frameId: main
actor: user
payload:
  selector: button[data-testid="checkout"]
  text: Checkout
  boundingBox:
    x: 420
    y: 640
    width: 120
    height: 40
artifactRefs:
  - artifact://screenshot-before-click
tags: []
```

## Example: Network Request Event

```yaml
id: evt_011
sessionId: sess_checkout
type: network.request
timestamp: 1781190001080
sequence: 11
pageUrl: https://shop.example.com/cart
frameId: main
actor: network
payload:
  requestId: req_abc
  method: POST
  url: https://shop.example.com/api/checkout
  resourceType: fetch
  headersRef: artifact://req-abc-headers
  bodyRef: artifact://req-abc-body
artifactRefs:
  - artifact://req-abc-body
tags:
  - json
```

- Should event IDs be generated by Browser Worker or Backend API?
